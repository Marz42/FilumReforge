# 20260423 改进执行清单

## 已完成

### 前端显示改进

- 看板已显示发布人姓名，且管理员可手动归档看板卡片与公告，使其不再出现在首页。
- 总览首页已收口为“看板 / 公告”在上、“待办事项 / 任务跟踪”在下；摘要卡仅保留待办与跟踪，并提供直达任务中心的跳转动作。
- 总览快捷入口已改为卡片式跳转，首页层次与对齐进一步收口。
- 任务中心主 tab 已调整为：待办事项、任务跟踪、发布任务、任务模板、备忘；历史任务已从主 tab 移除并并入“任务跟踪”视图，同时保留旧 `?tab=history` 兼容。
- 浏览器推送与 PWA 卡片已从消息中心拆出，统一进入新的“设置”模块，属于通用模块，所有登录用户可见。
- 登录页已取消预填 `admin@example.com` 和默认密码。

### 功能性改进

- 任务模板已支持删除，但仅允许删除“从未实例化过”的模板。
- 多设备 / 多浏览器推送链路已经成立：同一账号的全部活跃浏览器订阅都会收到 Web Push；设置页文案已明确该行为，运行时公钥仍通过 `GET /api/v1/push-subscriptions/config` 下发。

### BUG 修复

- 已修复“编辑模板后点击更新出现 500”的问题：根因是已有实例的模板在替换步骤结构时会破坏历史运行态。当前策略调整为：
	- 已有实例的模板仍允许修改名称、描述等元数据。
	- 已有实例的模板不允许继续修改步骤结构，前端会锁定结构设计器并提示新建模板版本。

## 下一批实施预案

### 岗位编辑器

1. 复用现有 `positions`、`profile_positions`、`reporting_lines`、字段权限与代理授权模型，不新增平行岗位体系。
2. 前端新增结构化岗位编辑器，默认拆为“基础信息 / 权限能力 / 组织可见范围 / 汇报关系 / 引用概览”五个区域，避免直接编辑 `metadata`。
3. 后端补一层岗位能力目录与引用摘要输出，明确哪些模块可引用岗位结果，例如任务模板默认负责人规则、字段权限模板、生命周期事件动作。
4. 首版不做拖拽式组织图编辑，先保证岗位定义、访问权限和跨模块引用可维护。

### 避免直接编辑 JSON

1. 治理原则：业务用户默认只使用结构化表单；JSON 仅保留为高级入口、导入工具或只读预览，不作为主交互面。
2. 优先改造顺序：人员工作台 / 档案治理中的 JSON 文本域优先，其次是模板设计器中的高级配置。
3. 落地方式：为每类 JSON 建立对应 schema、表单组件和校验提示；提交前统一转回后端现有 payload 结构，避免破坏 API。
4. 过渡期保留“查看 JSON”能力，便于排查与迁移，但默认折叠到高级区，不暴露给普通操作路径。

### 任务模板前端迭代方案

1. 将当前单页高密度交互拆成六个稳定视区：模板清单、基本信息、步骤设计、实例运行态、调度管理、高级 JSON。
2. 已有实例的模板默认进入“受保护模式”：结构设计只读，主动作改为“复制为新版本”，减少用户在同一页里同时处理历史实例和结构变更的心智负担。
3. JSON 导入 / 预览继续保留，但迁入高级抽屉；常规业务路径只暴露结构化表单、依赖选择、负责人规则和实例快照。
4. 调度、实例、版本操作应从模板基本信息中拆开，避免把创建、编辑、实例观察、排程和危险操作堆在同一屏内。
