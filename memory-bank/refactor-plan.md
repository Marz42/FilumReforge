# Project Filum 重构方案

**版本**: v1.2  
**状态**: 已确认并进入执行；Step 1 / Step 2 已完成，等待用户验测  
**适用范围**: 基于当前 Phase 5 已完成基线，对前端信息架构、部分领域模型、页面边界与权限分层进行重构规划

## 1. 重构背景

当前仓库已经完成用户、组织、档案治理、任务协同、模板、审批流、消息中心、知识库、AI Router、浏览器推送等能力，但信息架构仍然更接近“按阶段堆叠的工作台”，和你现在明确的产品结构还有几处偏差：

1. 首页仍是“仪表盘”，而不是以“总览”承接日常工作入口。
2. 任务能力已经很多，但尚未按“任务中心”重新分层为模板、发布、待办、跟踪、历史、备忘。
3. “审批中心”当前偏向流程引擎操作台，不符合“上下级逐级汇报 / 逐级传达”的业务定位。
4. “用户管理”和“档案管理”在产品层面应合并，但当前仍是两个独立页面。
5. 看板、公告、消息、任务提醒之间还缺少统一的信息入口与联动规则。

本方案的目标不是推倒重写，而是**在保留现有 Phase 5 可用底座的前提下，重构模块边界、页面入口、权限可见性与少量领域模型**，让系统的真实结构和你的业务模型对齐。

## 2. 重构目标

### 2.1 模块层级

系统重构为两类模块：

- **通用模块**：所有登录用户可见，但模块内的部分动作仍按角色、组织关系、特殊授权细分控制。
- **特殊模块**：只有指定权限组可见、可访问。

### 2.2 目标模块结构

#### 通用模块

1. **总览**
2. **任务中心**
3. **知识库**
4. **汇报中心**
5. **消息中心**

#### 特殊模块

1. **档案管理 & 用户管理**
2. **部门管理**

## 3. 重构原则

1. **保留现有底座，不做大爆炸重写**  
   优先复用现有 `tasks`、`task_templates`、`workflow_*`、`notification_*`、`attachments`、`profiles`、`departments`、`documents` 等模型与服务。

2. **先改信息架构，再改业务页面，再补领域模型**  
   先统一导航、入口、模块边界，再逐步下沉总览、任务中心、汇报中心的新能力。

3. **流程仍由服务层兜底**  
   前端只做可见性与操作入口控制；真正的权限、组织范围、字段裁剪、状态流转、提醒触发仍由后端服务保证。

4. **归档数据不可改写**  
   看板历史、公告历史、历史任务、汇报历史都按“归档后不可编辑”的原则设计。

5. **每一步都必须有测试出口**  
   所有重构批次都必须覆盖迁移 / 服务 / API / 前端回归，且在用户确认前不继续进入下一批。

## 4. 当前实现到目标结构的映射

| 当前实现 | 目标模块 | 处理方式 |
| --- | --- | --- |
| `HomeView` / `dashboard` | 总览 | 替换，不保留“仪表盘”概念 |
| `TasksView` | 任务中心 | 保留任务核心能力，重组成多标签工作台 |
| `TaskTemplatesView` | 任务中心 | 并入任务中心的“任务模板”子模块 |
| `ApprovalsView` | 汇报中心 | 不再以“审批中心”作为对外概念；审批引擎转为底层能力 |
| `MessagesView` | 消息中心 | 保留，强化用户隔离与联动提醒 |
| `KnowledgeBaseView` | 知识库 | 基本不改 |
| `UsersView` + `ProfilesView` | 档案管理 & 用户管理 | 合并为统一的人事管理工作台 |
| `DepartmentsView` | 部门管理 | 保留，补少量部门能力配置 |

## 5. 目标信息架构

## 5.1 通用模块

### A. 总览

总览将替代当前仪表盘，作为所有登录用户的默认入口，包含四个区域：

1. **看板**
2. **公告**
3. **当前任务**
4. **任务中心快捷入口**

#### A1. 看板

**业务定位**

- 展示当前用户所属组织树范围内的共享卡片。
- 适合部门、团队、公司层级的短周期信息同步。

**可见范围**

- 用户能看到自身所在组织路径上的看板卡片。
- 例如用户属于“公司 / 第一部门 / 第一小组”，则应看到：
  - 公司级卡片
  - 第一部门级卡片
  - 第一小组级卡片

**核心规则**

- 卡片需显示：
  - 发布人
  - 主题
  - 内容
  - 到期时间
- 每个用户同时最多发布 **2** 张有效卡片。
- 默认有效期 **7 天**。
- 到期后自动归档到历史存档。
- 历史存档不可编辑。

**建议数据设计**

- 新增 `board_cards`
- 新增 `board_card_archives`

建议字段：

- `id`
- `scope_department_id`
- `author_user_id`
- `author_profile_id`
- `title`
- `content_md`
- `published_at`
- `expires_at`
- `status` (`active` / `archived`)
- `archived_at`

设计说明：

- 活跃卡片放在 `board_cards`。
- 归档时写入 `board_card_archives` 快照，归档表禁止更新和删除。
- “每用户最多 2 张活跃卡片”由服务层校验，必要时加部分唯一约束或数据库检查辅助。

**提醒联动**

- 看板本身不做强推送默认渠道。
- 仅在“被 @ 指向”“高优先级置顶”“部门强提醒”这类后续增强场景考虑接入推送。

#### A2. 公告

**业务定位**

- 面向全公司。
- 只允许行政和 IT 部门发布。
- 用于发布长期有效且正式的信息。

**核心规则**

- 清晰显示：
  - 发布部门
  - 标题
  - 正文
  - 发布时间
- 默认不自动过期。
- 仅发布部门可手动撤下。
- 撤下后进入历史存档。
- 历史存档不可编辑。

**关键设计决策**

这里不建议把“行政 / IT 部门”写死为部门名称，而是新增**部门能力标记**，否则后续组织名称变更会导致逻辑脆弱。

**建议数据设计**

- 新增 `announcements`
- 新增 `announcement_archives`
- 新增 `department_capabilities` 或等效字段

建议能力项：

- `can_publish_announcement`
- `can_publish_org_task`
- `can_manage_templates`

**提醒联动**

- 新公告发布后，向全体可见用户写入系统消息。
- 是否同时强制 Web Push，建议作为后续开关能力，而不是默认硬编码。

#### A3. 当前任务

**业务定位**

- 面向当前用户显示“与我相关”的任务。
- “相关”的定义不是仅我是执行人，而是任务路径上出现过我、当前流转到我、我被抄送、我被要求关注或我作为发起人。

**页面结构**

分为上下两栏：

1. **待办事项**
   - 当前流转到我的任务
   - 按到期时间、紧急程度排序
2. **任务跟踪**
   - 与我相关的任务
   - 展示当前流转到哪里、卡在哪个节点

**提醒联动**

- 当任务正式流转到当前用户时，触发提醒。
- 提醒复用现有通知总线，默认走消息中心，已订阅用户进入浏览器通知。

**实现策略**

- 不重写核心任务模型。
- 复用现有 `tasks`、`task_logs`、`task_watchers`、`workflow_instances`、`workflow_step_runs`。
- 在服务层新增“我的待办投影”和“我的跟踪投影”查询接口。

#### A4. 任务中心快捷入口

- 总览只显示摘要与快捷入口，不承载复杂操作。
- 详情操作统一跳转到任务中心，避免首页过重。

### B. 任务中心

任务中心是“当前任务”的完整工作台，保留现有任务协同详情、活动时间线、负载概览，并重组为六个核心子模块：

1. **任务模板**
2. **发布任务**
3. **待办事项**
4. **任务跟踪**
5. **历史任务**
6. **备忘**

#### B1. 任务模板

**权限**

- 系统管理员
- 行政部门
- 部门负责人
- 其它被授权可管理模板的部门 / 用户

**设计方向**

- 继续复用现有 `task_templates`、`task_template_steps`、`task_schedules`。
- 把当前独立的 `TaskTemplatesView` 合并为任务中心的子页签。
- 模板的目标是显式描述任务流转步骤、参与角色、前后置关系和交付物。

#### B2. 发布任务

**权限**

- 管理层
- 行政
- 被授予“可发布组织任务”的部门 / 用户

**设计方向**

- 继续复用现有任务创建、模板实例化、审批触发能力。
- 前端入口从“任何人都能看到创建表单”调整为“所有人都能进入任务中心，但只有有权人能看到发布操作”。

#### B3. 待办事项

- 只显示流转到当前用户的任务。
- 默认排序：
  1. 紧急程度
  2. 到期时间
  3. 创建时间

#### B4. 任务跟踪

- 显示与当前用户相关的任务及当前所处节点。
- 重点保留：
  - 活动时间线
  - 评论留痕
  - 关注人 / 抄送
  - 当前步骤 / 审批节点

#### B5. 历史任务

- 已完成任务只读归档。
- 默认不允许从历史任务直接编辑；如需复用，应通过“基于历史任务生成模板 / 新任务”的方式处理。

#### B6. 备忘

当前仓库中还没有“备忘”领域模型，建议第一版按**个人备忘**实现，不直接并入消息中心。

**建议数据设计**

- 新增 `task_memos`

建议字段：

- `id`
- `owner_user_id`
- `related_task_id`（可空）
- `content`
- `is_pinned`
- `created_at`
- `updated_at`

这样既可以支持纯个人备忘，也可以支持“与某个任务关联的私人工作记录”。

### C. 知识库

- 当前先不改产品定位。
- 只做导航和命名上的一致性收口。
- 与其它模块的联动重点保留：
  - SOP 可被任务模板引用
  - `@系统` 可查询知识库

### D. 汇报中心

这里建议明确：**对外产品概念改为“汇报中心”，但底层继续保留现有 workflow engine 作为审批 / 流程引擎能力**。

原因是：

- 你现在想要的是“上下级之间的逐级沟通和流转”，而不仅是一个审批操作台。
- 但当前仓库已经有 `workflow_definitions`、`workflow_instances`、`workflow_step_runs`，完全推翻成本太高。

因此建议采用“两层结构”：

1. **产品层**
   - 汇报中心：面向用户的逐级向上汇报、逐级向下传达
2. **引擎层**
   - Workflow Engine：继续作为审批、会签、或签、打回、驳回的底层实现

#### D1. 汇报中心的能力划分

1. **向上汇报**
   - 员工向直属上级发起
   - 可逐级上推
   - 可附带任务、附件、说明
2. **向下传达**
   - 管理者逐级向下布置、传达
   - 可要求确认收到、要求反馈结果、要求生成任务

#### D2. 建议数据设计

建议新增独立汇报域，而不是直接把所有事情都塞进审批实例：

- `report_threads`
- `report_entries`
- `report_actions`
- `report_targets`

建议字段方向：

- `direction`：`upward` / `downward`
- `source_user_id`
- `current_handler_user_id`
- `scope_department_id`
- `status`
- `subject`
- `body_text`
- `related_task_id`
- `related_workflow_instance_id`

设计思路：

- 汇报中心承载“沟通与逐级处理”的主语义。
- 真正需要审批动作时，再挂接 `workflow_instance_id`。
- 这样不会把全部汇报都强行解释成审批。

### E. 消息中心

消息中心本轮不改定位，但要在重构中明确三件事：

1. **收件箱按用户严格隔离**
2. **与任务中心、汇报中心、公告、系统消息联动**
3. **有变动和新消息时触发提醒**

**设计要求**

- 用户只能查看自己的消息和自己的回执状态。
- 来自总览、任务、汇报、公告的提醒都统一沉淀到消息中心。
- 消息详情应显示来源模块和来源对象标识，便于回跳。

## 5.2 特殊模块

### F. 档案管理 & 用户管理

当前 `UsersView` 和 `ProfilesView` 需要合并为一个统一的人事管理工作台，面向 HR 与管理员。

#### 设计目标

- 一个入口同时管理：
  - 用户账号
  - 人事档案
  - 岗位与汇报关系
  - 生命周期事件
  - 字段权限可见结果

#### 建议页面结构

1. **人员列表**
2. **账号信息**
3. **档案信息**
4. **岗位 / 汇报**
5. **生命周期**
6. **权限视图**

#### 处理策略

- 后端保留 `users` 与 `profiles` 的分层，不做物理合表。
- 前端通过聚合接口和统一页面做产品层合并。
- 这样能保留现在已经稳定的认证、档案、权限服务边界。

### G. 部门管理

- 仍然只允许管理员访问。
- 页面主体不变。
- 需要补充的唯一重点是“部门能力配置”，以支持：
  - 哪些部门可发公告
  - 哪些部门可发组织级任务
  - 哪些部门可维护模板

## 6. 权限与可见性方案

## 6.1 顶层导航权限

| 模块 | 默认可见对象 | 备注 |
| --- | --- | --- |
| 总览 | 全部登录用户 | 入口统一 |
| 任务中心 | 全部登录用户 | 但模板 / 发布操作需额外权限 |
| 知识库 | 全部登录用户 | 继续沿用现有文档可见性 |
| 汇报中心 | 全部登录用户 | 但动作按上下级关系与授权控制 |
| 消息中心 | 全部登录用户 | 必须用户级隔离 |

## 6.2 模块内动作权限

| 动作 | 权限方案 |
| --- | --- |
| 发布看板卡片 | 登录用户可发，但受“同一用户最多 2 张活跃卡片”限制 |
| 发布公告 | 具有公告发布能力的部门成员 |
| 创建模板 | 管理层 / 行政 / 部门负责人 / 特殊授权用户 |
| 发布任务 | 管理层 / 行政 / 特殊授权用户 |
| 发起向上汇报 | 登录用户 |
| 发起向下传达 | 管理者 / 特殊授权用户 |

## 7. 后端重构策略

- `TaskTemplateService`
- `WorkflowEngineService`
- `MessageCenterService`
- `ProfileService`
- `OrganizationRelationService`
- `access_control.py`

## 7.2 计划新增 / 调整的服务

### 新增服务

1. `OverviewService`
2. `BoardService`
3. `AnnouncementService`
4. `TaskInboxService` 或在 `TaskService` 内新增投影查询
5. `ReportCenterService`
6. `PeopleAdminService`
7. `DepartmentCapabilityService`
8. `TaskMemoService`

### 需要重构的现有服务

1. `TaskService`
   - 区分“待办投影”“跟踪投影”“历史任务投影”
2. `WorkflowEngineService`
   - 从“直接对外审批中心”改为“供汇报中心和任务中心复用的底层引擎”
3. `MessageCenterService`
   - 补来源对象与回跳信息

   - 明确哪些事件默认触发消息、哪些事件进入 Web Push

1. `/overview`
   - 总览聚合接口

3. `/announcements`
   - 公告 CRUD、撤下、历史查询
4. `/task-center/*`
   - 待办、跟踪、历史、备忘聚合接口
5. `/reports/*`
   - 汇报中心接口
6. `/people/*`
   - 人员管理聚合接口
7. `/department-capabilities`
   - 部门能力配置接口

## 8. 前端重构策略

## 8.1 导航与路由

建议将当前路由重组为：

- `/overview`
- `/task-center`
- `/knowledge-base`
- `/reports`
- `/messages`
- `/people`
- `/departments`

并保留兼容跳转：

- `/dashboard` -> `/overview`
- `/tasks` -> `/task-center`
- `/approvals` -> `/reports`
- `/users`、`/profiles` -> `/people`

## 8.2 页面层拆分

### 总览

- 新建 `OverviewView.vue`
- 替换 `HomeView.vue`

### 任务中心

- 优先保留 `TasksView.vue` 的列表 / 看板 / 甘特图 / 时间线 / 负载能力
- 把 `TaskTemplatesView.vue` 并入 `TaskCenterView.vue`
- 新增待办、跟踪、历史、备忘标签页

### 汇报中心

- 重写 `ApprovalsView.vue` 为 `ReportsView.vue`
- 审批定义管理降为管理入口，不再是面向普通用户的主视觉

### 人员管理

- 新建 `PeopleManagementView.vue`
- 把 `UsersView.vue`、`ProfilesView.vue` 的信息合并展示

### 消息中心

- 保留 `MessagesView.vue`
- 新增来源类型、来源对象、回跳入口、未读聚合提示

## 9. 数据迁移与兼容方案

1. **路由兼容**
   - 旧地址先保留重定向，避免已有使用路径直接失效。

2. **服务兼容**
   - 旧任务、审批、用户、档案 API 在过渡期继续存在。
   - 新页面优先调用新聚合 API。

3. **数据兼容**
   - 任务、模板、审批、消息、档案主表尽量不做破坏性迁移。
   - 新需求优先以增表或扩展字段实现。

4. **归档兼容**
   - 看板和公告历史采用新表快照，避免污染现有业务表。

## 11. 风险与控制措施

| 风险 | 说明 | 控制措施 |
| --- | --- | --- |
| 审批中心直接改名导致语义混乱 | 现有流程引擎是“审批引擎”，新产品概念是“汇报中心” | 保留 workflow 作为引擎层，新建 report 领域承载产品语义 |
| 公告发布权限被写死 | 直接按部门名称判断不稳定 | 用部门能力配置替代写死名称 |
| 任务中心过大 | 把过多操作塞进一个页面会继续膨胀 | 用标签页 + 聚合接口拆分，保留现有明细面板 |
| 看板 / 公告归档被误改 | 业务要求归档不可编辑 | 采用归档快照表并限制更新 |
| 通知泛滥 | 所有变更都推送会打扰用户 | 明确默认提醒规则，区分站内消息和浏览器推送 |

## 12. 建议的首批执行优先级

建议按以下顺序进入真正开发：

1. **Step 1 壳层导航重构**
2. **Step 2 总览模块**
3. **Step 3 任务中心**
4. **Step 4 汇报中心**
5. **Step 5 人员管理合并**
6. **Step 6 消息联动收口**

原因：

- 先统一入口和命名，后续页面改造才不会反复返工。
- 总览和任务中心是最高频入口，应优先对齐。
- 汇报中心会影响较大的领域语义，应在导航和任务中心稳定后再落地。

## 13. 本方案中待你重点审阅的决策点

1. **公告发布权限是否接受“部门能力配置”方案，而不是写死行政 / IT 部门名称**
2. **汇报中心是否接受“独立 report 领域 + 复用 workflow engine”方案**
3. **备忘第一版是否接受“个人备忘，可选关联任务”的轻量设计**
4. **总览中的看板是否第一版只做发布 / 查看 / 自动归档，不做评论 / 点赞 / 互动**
