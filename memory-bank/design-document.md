**目标受众**: 产品、架构、前后端研发、后续接手开发者

## 1. 产品定位

Project Filum 面向 50-100 人规模企业，目标是在一个**模块化单体**系统内统一承载：

- **人**：账号、组织结构、人事档案、权限与授权
- **事**：任务、模板、审批、流程流转与协同留痕
- **信息**：消息通知、确认回执、浏览器推送、AI 指令入口

当前仓库已经完成 **Phase A / 文档与脚手架基线**、**Phase 1 / Foundation**、**Phase 2 / Collaboration & Stats**、**Phase 3 / HR Governance & Org Modeling**、**Phase 4 / Workflow Engine & Messaging**、**Phase 5 / Knowledge, AI Router & Experience**。  
本文件描述的是**产品目标与后续设计边界**；当前真实实现情况以 `memory-bank/architecture.md` 和 `memory-bank/progress.md` 为准。

## 2. 设计原则

### 2.1 模块化单体优先

- 明确反对过早拆分微服务。
- 用清晰的代码模块边界、服务抽象和数据库约束解决复杂度，而不是靠部署单元切碎系统。

### 2.2 抽象先于直连

- 业务层不直接操作对象存储、邮件、WebSocket、Web Push 或 LLM 平台。
- 外部能力统一收口到 service / adapter / worker 边界。

### 2.3 HR 数据安全优先

- 人事档案同时包含普通信息与高度敏感信息。
- 权限不能只停留在页面或接口级；必须补齐**字段级权限**与**关系型授权**。

### 2.4 角色、组织关系、字段策略三层叠加

- **角色级**：Admin / HR / Employee 控制全局功能边界。
- **组织关系级**：部门负责人、直属 Leader、虚线汇报关系、代理授权决定数据范围。
- **字段级**：不同身份对同一档案字段具备不同的查看 / 编辑权限。

### 2.5 工作沟通必须可追溯

- 与任务直接相关的讨论、补充说明、附件必须绑定到 `task_comments`。
- 不建设脱离任务上下文的“工作聊天系统”。
- “消息中心”承担的是通知、回执、审批提醒、系统消息，不替代任务协同留痕。

### 2.6 AI 是意图路由器，不是业务真相

- `@系统` 与 `/` 的本质是自然语言入口。
- LLM 负责理解用户意图、决定调用哪个工具、组织最终回复。
- 真实业务数据仍由后端服务和数据库提供。

## 3. 目标能力蓝图

### 3.1 组织与人事

- 一人一档：每个用户关联唯一主档案。
- 全生命周期：覆盖入职、转岗、晋升、奖惩、离职、返聘等事件。
- 多岗位关系：支持一人多岗、兼职、代理岗、跨部门挂载。
- 汇报关系：支持直属汇报和虚线汇报。
- 高敏字段：薪资、绩效、奖惩、晋升意见等必须具备字段级权限。
- 离职规则：HR 可以标记离职或停用账号，但不做物理删除。

### 3.2 用户与授权

- `Admin`：平台级最高权限，负责账号、系统配置与异常兜底。
- `HR`：负责档案创建、修改、生命周期维护，但不能删除员工实体。
- `Employee`：默认最小权限，只能访问自身或被授权范围内的数据。
- `Leader`：不是必须的独立全局角色，可由部门负责人、汇报线或临时授权推导获得。
- 代理 / 授权：领导休假或临时不在岗时，可把审批 / 任务处理权限委托给其他人。

### 3.3 事务引擎与协同

- 任务基础流转：`Todo -> Doing -> Review -> Done`
- 前后置依赖：任务 B 可以依赖任务 A 完成后再触发。
- 模板 / SOP：入职、离职、采购、报销等高频事务支持模板化生成。
- 模板运行态：高频事务模板需要支持按依赖逐步激活，而不是一次性暴露全部执行任务。
- 多人步骤协作：同一步骤需要支持多人扇出处理，并按模板配置的 `all` / `any` 汇聚规则推进到下一步。
- 审批流：支持串行审批、会签、或签、驳回、打回重做。
- 协同留痕：评论、日志、附件、状态变更统一归档。
- 定时任务：支持提醒、周期任务、模板定时实例化。

### 3.4 任务视图与使用体验

- 列表视图：适合日常筛选、统计与批量处理。
- 看板视图：适合持续流转与跨状态观察。
- 甘特图视图：适合周期较长、依赖关系明确的事务。
- 任务详情页需要同时展示：
  - 当前状态
  - 负责人 / 抄送人
  - 时间线
  - 评论与附件
  - 前置 / 后置关系

### 3.5 消息中心与通知总线

- 消息发送统一经过 `NotificationService.send(message_obj)`。
- 支持消息附件、投递状态、确认回执、已读 / 确认状态。
- 通知渠道按适配器接入：
  - Email
  - WebSocket
  - Web Push
  - 未来可扩展企微 / 钉钉
- 任务相关讨论仍然保留在 `task_comments`，不迁移到消息中心。

### 3.6 知识库与 AI Router

- 文档知识库使用 Markdown 管理制度与 SOP。
- 文档切块、嵌入、检索使用 PostgreSQL + `pgvector`。
- `@系统` / `/` 作为统一命令入口。
- LLM 通过 Tool Calling 调用后端工具，不直接拼接 SQL 或任意代码。

### 3.7 前端界面

- 主要形态为浏览器后台管理界面。
- 重点支持：
  - 表格、表单、时间线、审批列表
  - 浏览器消息推送
  - 后续 PWA 安装与桌面快捷入口
- 当前重构方向：
  - 顶层信息架构改为**通用模块 / 特殊模块**
  - 通用模块包括：总览、任务中心、知识库、汇报中心、消息中心、设置
  - 特殊模块包括：档案管理 & 用户管理、部门管理
- 模板设计器后续增强方向：
  - 首版采用结构化表单设计器，提供步骤列表、依赖选择、执行规则与汇聚规则配置
  - 保留 JSON 预览 / 导入作为高级入口，但不再要求业务用户直接编写步骤 JSON
  - 完整拖拽式流程图不是当前首版目标

## 5. 非目标与约束

- 不建设独立 IM / 聊天系统。
- 不把 LLM 当成业务主存储或审批主逻辑。
- 不用 LangChain。
- 不把薪资、绩效等高敏数据简单塞进“所有管理者都能看”的粗粒度权限模型里。
- 不使用“删除员工”作为标准离职流程。

## 7. 后续增强方向

当前建议的后续增强重点不再是“补齐 Phase 5”，而是：

1. **重构与测试强化**
   - 梳理服务边界、降低页面复杂度
   - 提升集成测试 / E2E 覆盖
2. **注册与账号开通**
   - 明确公开注册、邀请注册或审批式注册的业务选择
3. **HR 流程自动化**
   - 生命周期事件联动模板 / 审批流
4. **结构化任务模板与多步骤协作**
  - 模板实例运行态
  - 逐步激活、多人扇出 / 汇聚
  - 结构化模板设计器
5. **消息与渠道深化**
   - 消息附件
   - Email / WebSocket 外部接入深化
   - Push 观测与失败处理体验增强

## 8. 测试与验收原则

- 每个阶段继续遵循 **模型先行 -> 服务层封装 -> API 暴露 -> 前端对接**。
- 引入 worker / adapter 的阶段，需要在服务层稳定后补齐异步执行器与渠道适配验证。
- 每一步都必须有明确测试出口：
  - 迁移与约束测试
  - 服务层规则测试
  - API 集成测试
  - 前端交互测试
  - 必要的 worker / adapter 冒烟测试
- 用户在阶段验收通过前，不进入下一阶段编码；阶段后的关键 follow-up 也需要补齐验证与文档。
