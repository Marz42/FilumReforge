## 前端方面

- 最上面的“我的待办、跟踪任务...”等统计信息卡片不需要了，“任务中心 默认聚焦跟踪、发布、模板与个人备忘，待处理事项保留为快速入口。”这一块整个上移到顶。

- 抽离“发布任务”： “发布任务”本质是一个动作（Action），而不是一个视图区块。建议将它从标签页中移出，变成一个全局的 “+ 建立任务” 悬浮按钮 (FAB) 或页面右上角的主按钮，点击后弹窗（Modal / Drawer）填写。这样在看任何一个列表时都可以顺手建任务。

- 重排标签： 待处理 应该被放到绝对的 C 位（第 1 个标签并作为默认项）。顺序建议改为：待处理 -> 任务跟踪（含历史） -> 备忘 -> 任务模板。

## 逻辑结构部分

- 任务逻辑重构。现在我们只有两种任务，一种是单步任务，对应临时发布的额外任务，一种是固定的任务流，从模板中扇出
- 单步任务流程如下：

```

阶段 1：派发与要约 (Draft -> Assigned)
动作： 领导创建任务，明确目标、交付物要求、截止时间，指定给员工 A。
审计点： 系统记录发起时间、初始要求。此时任务并未真正开始计时，因为契约还没达成。
防抵赖意义： 领导不能在后续偷偷修改原始要求而不留痕迹。
阶段 2：双向确认 / 握手 (Assigned -> Accepted / Rejected)
这是目前绝大多数协同软件缺失，但对“责任到人”极其重要的一环。
动作： 员工 A 收到任务后，不能自动变成 Todo。员工必须做一个动作：“接受任务 (Accept)”。
如果员工认为不合理： 比如目标不清晰、时间不够、或者自己不是合适的人选，员工必须有权点击**“退回/协商”**，并填写理由。此时任务打回给领导重新调整。
防抵赖意义： 一旦员工点击了“接受”，就意味着他确认理解了要求，并承诺在这个截止时间内完成。后续如果延期或做差了，员工无法再说“我当时就没看懂”或“时间本来就不够”。
阶段 3：执行与透明 (Accepted -> Doing)
动作： 员工开始工作。
审计点： 记录 started_at。如果是长周期任务，系统应强制要求员工在任务里定期（如每天/每周）更新进度或标记 Block（受阻）。
执行力体现： 领导在 Tracking 视图里，看的是这个任务有没有按预期推进，有没有及早暴露风险，而不是盲等死线到来。
阶段 4：提交交付与自证 (Doing -> Pending Review)
动作： 员工完成任务，点击“提交验收”。
审计点： 强制员工填写**“交付说明”或附上“交付物”**。记录 submitted_at。
防抵赖意义： 员工声称“我做完了”，必须拿出证据。这也是锁定“按时交付”的关键时间戳，避免领导太忙没空看，导致员工背上“延期”的黑锅。
阶段 5：验收与评价 (Pending Review -> Done / Rework)
动作： 领导验收员工的交付物。
分支 A（验收通过）： 确认满足要求，选择任务完成质量评价，任务变为 Done。记录 completed_at。
分支 B（验收不通过 / 打回返工）： 发现交付物不合格，必须允许逆向流转（这是对你之前单向状态机的最大修正）。领导填写驳回理由，任务退回给员工（状态回到 Doing 或产生一个 Rework 状态）。
执行效果体现： 返工次数（Rework count）是衡量该员工“一次做对能力（执行质量）”的硬性指标。返工越多，执行效果越差。

```

- 多步任务逻辑如下：


```

以下是针对多责任人工作流的 5 项核心架构设计：
1. 拆分“宏观契约”与“微观契约” (Instance vs. Node)
必须在数据模型上把“整个流程”和“具体步骤”严格切分开。
宏观契约（Workflow Instance / 流程实例）：
代表整个“接力赛”。
归属： 必须有一个唯一的 Owner（通常是发起人）。
SLA： 拥有全局的截止时间（比如整个新员工入职流程必须在 7 天内跑完）。
视图体现： 这个实例应该出现在发起人的 Tracking（跟踪）列表里。发起人对最终结果负责。
微观契约（Task Node / 节点任务）：
代表接力赛中的“某一棒”。
归属： 每激活一个节点，系统就在底层生成一个独立的真实 Task，且只有一个唯一的执行人 Assignee。
SLA： 拥有局部的节点截止时间（比如 IT 部门分配电脑，必须在接到通知的 24 小时内完成）。
视图体现： 这个具体的 Task 会出现在当前负责人的 Inbox（待处理）列表里。
2. 严密的“交接棒”协议（Handover Protocol）
上一个节点的 Done 不能理所当然地变成下一个节点的 Doing。这中间必须有“防扯皮”的交接机制。
场景设定： A（内容撰写）完成后，流转给 B（法务审核）。
输入即前置输出（Context Passing）： B 的任务表单中，必须强制关联 A 的交付物快照。系统记录：“这是 A 提交的最终版本”。
防坑机制（Reject Upstream）： B 收到任务时，状态不能直接是 Doing。B 必须先进行**“验收入站”**。
如果 A 写的文章完全文不对题，B 有权在自己的节点点击**“打回至上游 (Reject to Previous)”**。
此时，B 的节点任务宣告中止（或者处于挂起），系统重新唤醒或生成一个 A 的返工任务。
第一性原理体现： 只有 B 点击了“接受（Accept）”，B 才正式开始承担当前节点的 SLA 倒计时。否则，耽误的时间算在 A 头上（返工耗时）。
3. 会签与并行的硬核解法 (Fan-out & Join)
如果在某个环节，领导要求“这篇文章需要 产品经理 X、法务 Y、公关 Z 三个人同时审核”。
错误做法： 建一个任务，执行人写 [X, Y, Z]。这必将导致“三个和尚没水喝”，最后没人审，且无法统计谁拖了后腿。
正确做法（Fan-out 裂变）： 你的现有架构里有 fan_out 机制，这是非常正确的。系统必须在底层同时生成 3 个完全独立的子任务，分别发到 X、Y、Z 三个人的 Inbox 中。
汇聚关卡（Join-mode 仲裁）：
如果规则是 Join_mode = ALL（全员会签）：必须等 3 个子任务全部变成 Done，下游的下一步任务才会被触发（Lazy Activation）。
如果规则是 Join_mode = ANY（或签/抢单）：其中一个人（比如 X）完成了，系统立刻自动将 Y 和 Z 的任务标记为 Cancelled（已取消），然后流程继续往下走。
第一性原理体现： 每个人只对分发给自己的那个单一任务负责。在数据库里，X、Y、Z 的处理时间、意见、是否延期，是三条清晰、独立的记录。
4. 跨级打回与溯源 (Deep Rejection)
工作流中最复杂的逻辑是长链路的打回。假设流程是：A(草稿) -> B(校对) -> C(设计) -> D(总监终审)。
D 发现设计图里的文字有严重错误（是 A 写的），D 怎么处理？
系统必须提供**“退回指定节点”的功能。D 不能只退给上一级 C，D 应该有权直接选择“退回给 A (草稿)”**。
责任链修复： 当退回给 A 时，流程状态被打破。A 重新修改后，工作流该怎么走？
常规严谨模式： 必须重新走一遍 A -> B -> C -> D，以保证每次修改都被沿线人员重新确认（防抵赖）。
灵活模式（通常需要人工介入）： A 改完后，由于不影响设计 C 的排版，A 可以直接在系统里申请“提交给 D 复核”。
建议： 在底层数据设计上，所有的跨级打回，都要视为当前工作流实例发生了一次“版本迭代”。A 重新提交的东西属于 V2，后续的所有节点都在基于 V2 重新生成微观任务。旧的微观任务作为历史存档，不可篡改。
5. 责任瓶颈的全局暴露 (Bottleneck Visibility)
既然是多人流转，就一定会有“卡壳”的时候。对整个流程负责的 Owner（发起人）需要极强的雷达。
在 task_center_service 提供给发起人的 Tracking（跟踪视图）中，绝不能仅仅显示状态为 In Progress。
必须透传当前卡点（Current Blocker）： 必须清楚地渲染出：“当前流程已耗时 5 天，卡在第 3 步【法务审核】，当前持棒人：法务Y，此节点已超时 12 小时”。
在前端，提供一个**“一键催办 (Ping/Nudge)”**按钮，其实质是向当前持棒人发送一条不可抹除的系统级催促日志。


- 说明：多步工作流并不是一种“特殊的任务”，而是**“一个协调器（Orchestrator） + N 个单步任务（Atomic Tasks）”**的组合。单步任务就像是“一块标准砖”，而工作流是一张“图纸”。图纸负责决定什么时候、把哪块砖放在哪里，但砖本身的硬度、尺寸（即任务的 DRI、SLA、验收防抵赖机制）是绝对一致的。你不仅不应该重写任务逻辑，反而应该极致地复用你现有的 TaskService。
