

### 一、 核心领域模型抽象 (Core Domain Model)

在新的设计中，我们将彻底废除全局硬编码的 `Todo -> Doing -> Review -> Done` 状态机制，将所有业务动作抽象为“图（Graph）”与“节点（Node）”。

#### 1. 核心实体的重新定义
*   **Workflow Template (工作流模板)**：由发起人定义或系统预设的“蓝图”。包含节点定义、流转规则配置。
*   **Workflow Instance (工作流实例)**：每次发起任务时，根据模板“深拷贝”生成的运行态对象。**快照原则**：一旦生成实例，其底层定义的结构将被快照固化，后续对模板的任何修改都不影响正在运行的实例。
*   **Node (节点)**：流转中的最小工作单元。所有节点本质相同，仅通过配置项产生行为差异。
    *   *Task Node (普通任务节点)*：要求输出明确交付物。
    *   *Approval Node (审批节点)*：要求输出布尔值（同意/驳回）及审核意见文本。
    *   *Notice Node (通知节点)*：静默节点，触达即完成，不阻塞主流程（用于自动抄送和知会）。
*   **Edge (边/路由)**：连接节点的路径，带有**条件判定（Condition）**属性。例如：“当节点A的结果为[同意]时激活”。
*   **Deliverable (交付物)**：挂载于节点之上的数据实体，可包含文本、表单、文件附件等。

#### 2. “Append-Only” 历史轨迹原则（极其重要）
DAG 的特性是“无环”，但现实业务中的“审批驳回”天然是一个环（Loop）。
**设计决策：我们不在图中实现物理回退，而是采用“追加式（Append-Only）”执行。**
如果节点 B 驳回给节点 A，引擎**不会**把状态机倒退回原来的节点 A，而是**动态克隆生成一个全新的节点 A (版本 2)**，并将其接入执行链路。
*   *收益*：整个流转记录是线性向前的，彻底解决“历史交付物被覆盖”和“审计追踪混乱”的问题。每次打回和重新提交的过程、附件、耗时都清清楚楚。

### 二、 节点微观状态机与流转引擎 (Execution Engine)

每个 `Node` 自身拥有独立的生命周期，引擎通过监听节点状态变更来驱动整个 Graph 运转。

#### 1. 节点级状态机 (Micro-State Machine)
*   `Pending (等待激活)`：前置依赖节点尚未完成。
*   `Activated (已激活)`：前置条件满足。此时系统触发 Webpush/邮件给处理人。
*   `Acknowledged (已确认)`：处理人点击了“收到”。（此时可作为 SLA 耗时考核的起点）。
*   `Completed (已完成/流转)`：用户提交了交付物/审核意见，并选择了一个“出口”。
*   `Terminated (强行终止)`：因分支条件未命中，或整个实例被撤销，导致节点被系统遗弃。

#### 2. 并发与汇聚的调度逻辑 (Fan-out & Join)
*   **并发 (Fan-out)**：当节点 A 完成后，其出口定义了同时走向节点 B 和 节点 C。引擎会将 B 和 C 的状态同时更新为 `Activated`。
*   **汇聚 (Join)**：节点 D 的启动条件依赖节点 B 和节点 C。引擎必须实现**“等待锁（Wait-All）”**机制。只有当引擎校验到 B 和 C 都处于 `Completed` 状态时，D 才会被激活。

### 三、 组织架构联动与“智能抄送”机制 (Smart Routing)

针对“越级派发时自动同步中间领导”的业务需求，设计如下：

#### 1. 链路计算触发时机
在用户填写完前端表格（指定了最终接收人）并点击“提交/发起”的那一刻，或在选择人员后通过防抖 API 调用后端计算。

#### 2. 最短汇报线算法意图
引擎查询底层的组织树与汇报线图：
1.  定位发起人节点（如总经理）和接收人节点（如基层员工）。
2.  向上溯源基层员工的直属部门汇报线，直到找到与总经理的重合点（通常是最高层或发起人自身）。
3.  提取路径上的所有管理节点（如：部门经理、总监）。

#### 3. 交互与生成控制
*   **人工干预优先原则**：算法计算出的名单，必须在前端以“知会人/抄送人”的标签列表形式**回显给发起人**。允许发起人点 `X` 移除某位领导（例如保密项目不想让中间人知道），或手动追加其他人。
*   **静默处理**：确认发起后，这些抄送人会被系统包装为 `Notice Node (通知节点)`，并行挂载在第一个工作节点旁边。他们会收到一条带有任务卡片的 Webpush，但该卡片仅有“查看”权限，无需点收到。

### 四、 前端“基于表格的 DAG”表达方案 (Adjacency List UI)

在不使用拖拽画布的前提下，如何用表格优雅地表达复杂的网状结构？

#### 1. 列设计意图
前端将采用“邻接表”的概念，展示一个动态行表格，每行代表一个节点配置：
*   **[序号/标识]**：如 步骤 1, 步骤 2 (Step ID)。
*   **[动作类型]**：下拉选择（任务交付 / 审批 / 归档）。
*   **[处理人]**：支持选具体人，或选动态角色（如：发起人的直属主管、上一环节处理人的部门经理）。
*   **[输入/交付要求]**：定义该环节需要交什么（提供富文本输入框配置要求，或勾选“需上传附件”）。
*   **[成功流转至]**：多选下拉框，选项为当前表格中的其他步骤序号。（配置 Fan-out并行的关键）。
*   **[失败/驳回流转至]**：仅审批动作出现。单选下拉框，通常指向之前的某个步骤，或选择“终止整个任务”。

#### 2. 前端验证屏障 (Topological Validation)
**这是表格设计最容易翻车的地方。** 用户很容易配出一个死循环或孤岛节点。
在提交之前，前端必须在内存中对表格数据进行“有向图拓扑排序校验”：
1.  **无头/无尾校验**：是否存在没有前置节点的节点（除了首节点）？是否存在没有后续出口且不是最终节点的节点？
2.  **死循环校验（静态解析）**：在正常流转路径（不考虑驳回路径）上，绝对不能出现 Step 2 指向 Step 1 的配置，必须保证从左到右/自上而下的流向。

### 五、 审计、日志与数据落盘 (Auditing & Persistence)

由于所有动作都被拆散到了 Node 级别，过去的“针对整个任务的单一时间线”将无法满足复杂的流转查看需求。

#### 1. 上下文强隔离
*   评论、内部备注、上传的附件不再挂载到 `Task` 实体上，而是强行绑定到 `Node Instance (节点实例)` 上。
*   **页面渲染意图**：当用户打开任务详情页，右侧/下方的时间线应以**“节点板块”**为单位进行渲染。
    *   *(板块)* **[步骤1: 需求编写] (已完成)** -> 展示当时的交付附件、耗时、该步骤内的评论。
    *   *(板块)* **[步骤2: 部门审批] (已驳回)** -> 展示领导当时的打回意见。
    *   *(板块)* **[步骤1-V2: 需求编写] (进行中)** -> 当前最新进展。

#### 2. 状态扭转的安全与幂等性
*   分布式系统或弱网环境下，用户可能连点两次“提交交付物”。
*   引擎必须在流转接口使用乐观锁（基于 `node_instance_version`）或分布式锁。一旦节点从 `Activated/Acknowledged` 变为 `Completed`，立即拒绝任何修改该节点的请求，确保下游节点不会被重复触发（Fan-out 重复执行灾难）。

### 六、 异常边界与极端情况防御 (Edge Cases & Resilience)

作为企业级架构设计，必须考虑到系统运转时的非理想状态：

1.  **“人员变动”引发的卡死节点 (The Ghost Assignee)**：
    *   *问题*：任务流转到节点 C，但处理人今天刚离职，或正在请长假，节点永远等不到 `Acknowledged`。
    *   *设计对策*：引入**“节点接管（Takeover）”机制**。部门管理员或系统管理员拥有特权视图，可以将卡在某人身上的运行态节点强制“转派（Reassign）”给另一人。记录特权操作审计日志。

2.  **审批无限打回防御 (Infinite Rejection Loop)**：
    *   *问题*：甲做完乙打回，甲再做乙再打回，形成拉锯战。由于我们采用的是 Append-Only 节点追加模式，这会导致数据库无限增长。
    *   *设计对策*：在实例维度设定一个 `max_iterations` 阈值（例如同一节点最多生成 5 次版本）。触发阈值时，自动挂起整个 Workflow，并通知发起人介入协调（系统层面转化为人工干预）。

3.  **消息丢失防抖 (Message Reliability)**：
    *   *问题*：节点激活后，调用 Webpush 失败（用户浏览器离线或中间件宕机）。
    *   *设计对策*：引擎状态流转不能强依赖通知触达。采用**“发件箱模式 (Outbox Pattern)”**，流转事务提交的同时，在本地库记录一条待发送消息。由后台异步 Worker 不断尝试推送，直到收到 ACK，同时结合站内的红点未读消息做双保险。

4.  **并发汇聚的“数据污染” (Data Race in Join)**：
    *   *问题*：Fan-in 汇聚时，节点 B 和 C 都在同一毫秒完成，同时触发引擎去激活节点 D。如果锁控制不好，D 可能会被激活两次。
    *   *设计对策*：依赖数据库的唯一索引或强一致性事务。激活 D 的前置动作必须包含对其所有入度（In-degree）节点的严格校验，使用类似 `SELECT ... FOR UPDATE` 锁住 D 所在的 Graph Instance。
