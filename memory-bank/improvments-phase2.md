# Project Filum 下一周期改进方案（Phase 2）

## 1. 文档定位

本文件用于承接当前仓库在 Step 7 收口之后的下一周期改进工作，目标不是重述历史阶段，而是基于当前真实代码、README 与 memory-bank 基线，给出一份可执行的后续推进方案。

本文件重点覆盖三个方面：

- 项目当前情况和已实现功能
- 前端改进方向
- 功能改进方向

说明：`memory-bank/improvements-2.md` 仅作为未收敛草稿参考，不作为当前实现事实来源，也不直接作为排期依据。

## 2. 当前项目情况总结

- 当前仓库已经完成 Phase A、Phase 1-5，以及重构 Step 1-7。
- 当前主线已经从“重构收口”转入“工作流 E 深化、回归、部署工程化与稳定性强化”。
- 生产部署产物已经具备，包括后端与前端 `Dockerfile.prod`、生产 Compose、Nginx 生产配置、`start-prod.sh` 与发布前校验脚本。
- memory-bank 主文档、根目录 README、前后端 README 与当前实现主线整体上是对齐的，没有发现“核心能力已缺失但文档宣称已完成”的明显断层。

- `scripts/check-release.sh` 发布前校验脚本

- 生命周期事件与模板/审批流仍缺自动联动

### 2.5 `workflow-refactor.md` 与当前实现的差距判断

`memory-bank/workflow-refactor.md` 提出的是一套更激进的图节点化工作流重构方案。它与当前项目并非完全脱节，但也不是“下一周期直接整体替换”的级别，而更适合被拆解为中长期演进方向。

#### 当前实现已经接近该方案的部分

- 当前任务模板已经具备模板、步骤、依赖、实例、步骤运行态四层模型，并且实例运行态带有快照视图。
- 当前系统已经支持 fan-out / join、步骤依赖、审核驳回目标、步骤重跑迭代，以及模板实例驱动的逐步激活。
- 当前通知链路已经具备“先落库、再异步投递”的基础形态，`notification_messages` / `notification_deliveries` + ARQ 已经是一个弱化版 outbox。

#### 当前实现与该方案的关键差距

- 当前系统仍然是“任务为主、步骤运行态为辅”的模型，而不是“节点实例为主、任务只是节点的一种表现”。
- 当前严格任务状态机 `Todo -> Doing -> Review -> Done` 仍然是任务中心、首页、消息来源回跳等前台能力的重要基础，不能在下一周期被直接废除。
- 当前评论、内部备注和附件上下文仍然绑定在 `Task`，不是绑定在节点实例或步骤运行态。
- 当前“已确认/Acknowledged”只存在于消息回执层，不存在于任务节点微状态层。
- 当前打回/驳回已经部分采用追加式 iteration，但还没有形成统一的 append-only 节点历史模型。
- 当前不存在 `Notice Node` 这一类静默节点，也没有“越级派发自动生成中间领导知会链”的智能抄送能力。

#### 合理性判断

- `workflow-refactor.md` 的方向总体合理，尤其是 append-only 历史、节点级上下文隔离、邻接表式前端表达、并发汇聚幂等控制，这些都切中了当前工作流 E 下一阶段真正会遇到的复杂度问题。
- 但它把“彻底废除全局任务状态机、完全改成图节点原生模型”作为起点，这对当前项目来说过于陡峭，会直接冲击任务中心、消息中心、总览页、任务详情、权限判断和现有测试基线。
- 因此更合理的落地方式不是重写，而是渐进吸收：保留 `Task` 作为用户可见工作项，把 `TaskTemplateStepRun` / 后续 node runtime 逐步提升为真正的流程控制层。

## 3. 下一周期总目标

下一周期不建议再以“大而全新模块”作为主线，而应聚焦三个目标：

1. 把现有工作台做得更清晰、更稳定，优先解决首页、任务中心、模板工作台的使用成本。
2. 把工作流 E 从“已可用”推进到“可管理、可回归、可部署”。
3. 把系统从“功能已成型”推进到“可持续上线与扩展”，补上稳定性、验证和发布工程化短板。

## 4. 前端改进方案

### 4.1 总览页重排与信息优先级重构

#### 现状判断

- 总览页已经具备看板、公告、待办、任务跟踪和快捷入口。
- 但页面首屏仍以看板/公告大卡片为主，用户最常用的任务指标和动作入口没有处于最强视觉层级。
- 空状态卡片占位仍偏大，容易稀释有效信息密度。

#### 改进目标

- 将“我现在要处理什么”置于首页第一优先级。
- 压缩空状态占位，提升首屏有效信息密度。
- 明确“统计概览”和“明细列表”的语义边界，减少重复命名。

#### 建议动作

- 将首页统计卡片提升到首屏顶部，优先展示待办数、跟踪数、未读消息数、待审批数。
- 看板/公告区域改为自适应高度；无数据时用紧凑空状态，不再保留大面积插画式占位。
- 将当前“待办事项”统计卡与下方“待办事项”列表进行命名区分，例如统计层改为“任务概览”或“我的工作”。
- 强化首页快捷入口与最近动作联动，允许从统计卡直接跳转到对应任务中心 tab 或消息中心筛选。

#### 预期产出

- 更清晰的首页视觉层级
- 更高的信息密度与更少的空白浪费
- 更直接的首页到任务中心转化路径

### 4.2 任务中心结构收敛

#### 现状判断

- 当前任务中心保留五个主 tab：待办、跟踪、发布、模板、备忘。
- 统计卡仍包含“历史任务”，但历史已并入跟踪语义，页面整体上还有进一步压缩与统一口径的空间。
- 用户真正高频使用的通常是跟踪、发布、模板和备忘，待办数据与首页待办列表之间存在功能重叠。

#### 改进目标

- 让任务中心更像“操作工作台”，而不是“所有任务入口的堆叠页”。
- 明确首页负责“快速处理”，任务中心负责“完整管理”。

#### 建议动作

- 评估将“待办事项”从任务中心主 tab 中下沉为首页/消息驱动入口，任务中心主 tab 收敛为：任务跟踪、发布任务、任务模板、备忘。
- 若短期不移除待办 tab，也应至少弱化其在页面中的优先级，避免与首页待办形成双入口竞争。
- 统一统计卡与 tab 的口径，避免保留已被合并但名称未调整的历史指标。
- 强化任务中心内部深链能力，例如从首页/消息中心带筛选条件进入特定 tab 和视图状态。

#### 预期产出

- 更少的认知切换成本
- 更一致的“首页处理、工作台管理”分工
- 更容易扩展后续模板与调度管理能力

### 4.3 模板工作台体验深化

#### 现状判断

- 模板页已经实现结构化设计器、JSON 导入、实例快照与已有模板编辑。
- 当前页面承担的职责较多：模板列表、步骤设计、实例化、实例运行态、JSON 高级入口都集中在一个工作台里。

#### 改进目标

- 保持“结构化编辑优先，JSON 作为高级入口”的设计原则。
- 降低模板页的单屏复杂度，让模板设计、运行态查看和实例管理各有稳定区域。

#### 建议动作

- 将模板工作台明确拆分为六个稳定视区：模板清单、基本信息、步骤设计、实例运行态、调度配置、高级 JSON。
- 对“已有实例的模板”强化结构锁定提示，默认引导新建版本，而不是继续直接修改结构。
- 为结构化设计器补齐前端校验：依赖闭环、步骤 key 重复、空 assignee rule、join_mode 与 assignment_mode 组合非法等。
- 吸收 `workflow-refactor.md` 的邻接表思路，把“成功流转至 / 驳回流转至 / 动作类型 / 动态处理人 / 输入要求”逐步显式化，而不是继续把复杂语义藏在松散 JSON 中。
- 在前端提交前增加拓扑校验：无头/无尾校验、正常流转路径无环校验、孤岛步骤校验。
- 为模板实例运行态增加更直接的步骤状态可视化与失败原因提示。

#### 预期产出

- 更低的模板设计门槛
- 更少的误操作和结构破坏风险
- 更适合承接生命周期事件联动和调度治理

### 4.4 人员工作台与治理类页面的结构化编辑改造

#### 现状判断

- 当前项目已经具备人员工作台、档案治理、岗位/汇报线/生命周期等能力。
- 治理型页面后续如果继续增加复杂规则，直接暴露原始 JSON 或将多类配置堆在单表单中，会快速提升维护成本。

#### 建议动作

- 将规则录入优先保持为结构化表单，不让业务用户直接维护 JSON。
- 推进岗位编辑器、可见范围、规则能力等工作台的结构化配置界面，为后续模板受理人规则和治理联动复用。
- 统一表单校验、只读态、字段级权限提示和危险操作确认逻辑。

### 4.5 前端工程质量改进

#### 建议动作

- 为总览页、任务中心、模板页补更多视图级单测与交互回归。
- 梳理大型页面中的副作用和请求状态，减少重复请求、隐藏耦合和手工同步状态。
- 为首页、消息中心、模板运行态补充加载骨架、错误态、空状态三套一致体验。
- 将只读校验命令与自动修复命令分离，日常回归优先使用 `oxlint` 与 `eslint` 只读检查。

## 5. 功能改进方案

### 5.1 工作流 E 深化为第一优先级

#### 目标

把当前“可设计、可实例化、可逐步激活”的模板能力，推进为“可版本化、可运维、可稳定发布”的工作流能力。

同时，将 `workflow-refactor.md` 中真正合理的部分吸收进现有工作流 E，而不是另起一套完全平行的新引擎。

#### 建议动作

- 补全模板管理动作：版本管理、归档/停用、复制模板、调度启停、调度历史与最近执行结果。
- 强化调度治理：显示下次执行时间、最近执行结果、失败重试提示和禁用原因。
- 完善模板实例与步骤运行态查询能力，便于前端和运维排查。
- 扩展 workflow E 的服务测试、API 测试、前端交互测试与发布前验证脚本覆盖。

#### 吸收 `workflow-refactor.md` 的建议方式

- 短期不废除 `Task` 状态机，而是把 `Task` 明确保留为用户工作项，把步骤运行态继续作为编排内核。
- 将当前 `TaskTemplateStepRun` 逐步向“节点运行实例”演进：补更明确的状态、结果、版本和历史语义。
- 统一“驳回/打回”的 append-only 规则，避免回写旧步骤记录，继续沿用 iteration 追加新运行记录。
- 为 fan-out / join 激活增加更严格的幂等保护和唯一性约束，避免并发情况下重复激活下游节点。
- 把模板运行态与任务详情进一步联动，让页面能按步骤板块展示历史，而不只是一串任务列表。

#### 预期收益

- 模板能力从“首批可用”进入“可持续运营”
- 为真实业务流程沉淀模板资产
- 减少模板配置错误带来的运行态问题

### 5.1A 节点化重构的中长期吸收路线

#### 第一层：在现有模型上补齐“节点感”

- 为步骤运行态补更细粒度状态，例如 `pending / active / completed / cancelled` 之外，评估增加“已通知”“已确认”等可选微状态。
- 将步骤级结果、驳回原因、重跑版本、当前阻塞依赖做成一等查询字段，而不是依赖页面端二次推导。
- 将任务评论/附件与 `template_step_run_id` 建立更强关联，逐步把“任务评论”升级成“步骤上下文评论”。

#### 第二层：补 Notice/抄送与组织链路智能路由

- 在现有 watcher / 抄送基础上，先实现“智能推荐知会人”，而不是一步到位引入完整 `Notice Node`。
- 基于现有组织树、汇报线和部门负责人关系，提供“发起人与最终执行人之间的中间管理链推荐”。
- 前端必须允许人工删除和追加推荐对象，保持人工干预优先。

#### 第三层：再评估是否进入真正的 Graph/Node 原生模型

- 只有在步骤级上下文、append-only 历史、智能抄送、并发幂等、运行态查询都稳定之后，才评估是否进一步抽象 `Task` 为 Node 的一种视图。
- 在此之前，不建议发起“彻底移除 Todo/Doing/Review/Done”的高风险重写。

### 5.2 生命周期事件与流程联动

#### 目标

把 HR 生命周期从“记录事件”推进到“事件驱动任务/审批流”。

#### 建议动作

- 入职事件自动触发入职模板任务
- 离职事件自动生成交接、回收、审批链
- 转岗/晋升事件驱动岗位、权限、流程模板、消息通知的联动更新
- 加入幂等保护、重复触发保护和人工补偿入口

#### 预期收益

- 将 HR 治理模块与事务引擎真正打通
- 提升系统业务闭环程度
- 为中型组织提供更有辨识度的核心能力

### 5.3 消息中心深化

#### 目标

把消息中心从“聚合通知箱”提升为“可回执、可观测、可扩展”的统一通知中台。

#### 建议动作

- 补消息附件模型与前后端展示
- 深化 Email / WebSocket 渠道接入，不再停留在最小实现
- 提供投递失败、过期订阅、重试结果等可观测信息
- 补充更细粒度筛选，例如按来源模块、回执状态、时间范围、投递渠道筛选
- 保持当前“先落库再异步投递”的基本模式，进一步向更明确的 outbox / retry 观测靠拢，而不是把通知成功与否耦合到主流程事务提交结果上。

### 5.4 注册与账号开通策略定稿

#### 目标

解决当前“只能由管理端建用户”的产品边界问题。

#### 建议动作

- 先做产品决策：公开注册、邀请制、审批制三选一，或分层组合
- 再补齐邀请 token/待审核状态/默认角色策略/审批链逻辑
- 前后端统一处理注册、激活、过期、撤销与登录联动

#### 说明

该项重要，但优先级应低于工作流 E 深化和生命周期联动；因为当前系统的核心辨识度已经不在“能不能注册”，而在“组织治理 + 流程协同”是否形成闭环。

### 5.5 部署、发布与稳定性工程化

#### 目标

把现有生产产物从“文件已具备”推进到“上线路径经过演练”。

#### 建议动作

- 在预发或目标主机上完整执行一次 `scripts/check-release.sh`
- 按 Ubuntu runbook 走一轮从拉取、迁移、构建、重启到验证的完整演练
- 明确发布失败回滚步骤与最小可恢复路径
- 为 worker、Push、RAG、模板调度补更多运行态日志与错误定位信息

### 5.6 测试与观测补强

#### 建议动作

- 补 workflow E、消息中心、生命周期联动的 API 集成测试
- 增加至少一条关键业务 E2E smoke：登录 -> 首页 -> 模板实例化 -> 任务推进 -> 消息回跳
- 补充错误事件、request id、异步任务失败日志的排查指引
- 将文档、README、runbook 与测试命令同步维护，避免再次出现“实现已存在但说明仍停在旧阶段”

## 6. 推荐推进顺序

### 6.1 第一阶段：前端收敛 + 工作流 E 治理

建议先完成以下事项：

- 总览页重排与空状态优化
- 任务中心结构收敛
- 模板设计器与实例运行态的前端校验增强
- 模板管理/调度治理能力补齐
- workflow E 相关回归补强

原因：这一阶段对现有可见体验改善最大，同时能直接降低后续功能扩展成本。

### 6.2 第二阶段：生命周期联动 + 消息中心深化

建议随后推进：

- 生命周期事件驱动模板/审批联动
- 消息附件
- 通知渠道可观测性与失败处理
- 智能抄送推荐与步骤上下文收口
- 相关 API / worker / 前端联动回归

原因：这一阶段会把目前各模块从“并列能力”推进到“业务闭环能力”。

### 6.3 第三阶段：注册策略与发布演练收口

建议最后推进：

- 注册/邀请/审批式开通策略定稿并实现
- 发布前一键校验脚本实际执行
- Ubuntu / Compose 生产部署演练和回滚预案固化

原因：这部分更偏工程与入口补齐，重要但不应打断前两阶段主线。

## 7. 验收标准

### 7.1 前端改进验收

- 首页首屏以任务与动作信息为主，不再被空状态大卡片主导
- 任务中心入口职责更清晰，tab 与统计口径一致
- 模板工作台结构更稳定，结构化设计器校验更完整

- 生命周期事件能够稳定触发模板任务或审批流
- 消息中心具备附件和更清晰的投递状态表达
- 发布脚本、部署 runbook 至少完成一次真实演练

### 7.3 文档验收

- `memory-bank/architecture.md` 同步新的真实能力与模型变化
- `memory-bank/progress.md` 仅在阶段验收通过后更新
- 根目录 README 与前后端 README 保持与当前主线一致

## 8. 本文件的结论

Project Filum 当前已经不是“功能尚未成型”的项目，而是一个能力面较完整、需要进入“体验收敛 + 功能联动 + 稳定性工程化”阶段的模块化单体平台。

下一周期最值得投入的，不是继续横向铺新模块，而是把已经做出的总览、任务中心、模板系统、消息中心、HR 生命周期治理真正打通，并把部署与验证链路补齐。只有这样，系统才会从“完成了很多功能”进入“形成稳定产品能力”。
